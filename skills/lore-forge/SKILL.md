# Lore Forge

## 概述 (Overview)

Lore Forge 旨在读取用户的简历文件，自动识别其中涉及的所有技术栈和知识点，并以此为基础，生成一套**教科书级别**的现代化自学文档网站。
它不仅仅是一个知识图谱或简单的总结，而是一本**详尽的教材**。它会从最基础的前置知识讲起，逐步深入到简历中提到的核心技术，最后辅以真实的面试题模拟。
生成的网站将采用**多文件结构**（按章节或模块拆分 HTML），并引入轻量级的现代前端依赖（如 Tailwind CSS、Alpine.js 等），以打造一个现代化、交互流畅且具有极高信息密度的工具型阅读体验。

## 工作流 (Workflow)

### 第一步：解析简历文件

根据用户提供的文件格式，选择对应的解析方式：

- **PDF / DOCX**: 在 skill 的 `scripts/` 目录下执行以下命令提取纯文本内容：
  ```bash
  npm install 2>/dev/null && npx tsx parse_resume.ts <file_path>
  ```
- **TXT / MD**: 直接使用读取工具获取文件内容。

### 第二步：知识体系构建与深度分析

从简历文本中提取所有技术知识点，并将其扩展为一个**完整的教科书知识体系**。不要仅仅罗列名词，必须梳理出完整的学习链路。按以下维度进行分类：

1. **编程语言与框架** — 如 Java、Python、React、Spring Boot 等
2. **系统与架构** — 如微服务、分布式系统、消息队列等
3. **数据库与存储** — 如 MySQL、Redis、MongoDB 等
4. **DevOps 与工具** — 如 Docker、K8s、CI/CD、Git 等
5. **计算机基础** — 如操作系统、网络、数据结构与算法等
6. **业务与领域知识** — 如支付系统、推荐系统等
7. **其他技能** — 如项目管理、软技能等

**挖掘前置知识（Prerequisites）**：这是至关重要的一步。对于每一个核心知识点，必须向下挖掘其依赖的基础知识。例如，提到 Spring Boot，必须先讲解 Servlet、Spring Core、IoC/AOP 等前置概念；提到 Redis，需要讲解内存存储原理、数据结构基础等。

### 第三步：生成教科书级别的章节结构

按照划分好的知识分类组织章节，每一章/节必须包含极其详细的教学内容：

- **章节概述**：通俗易懂的背景介绍，解决“为什么需要这个技术”的问题。
- **前置知识精讲**：在讲解核心技术前，先用充足的篇幅把依赖的基础知识讲透。
- **核心知识点教学**：深入讲解该技术的底层机制、工作原理和核心组件。
- **高频面试题模拟**：在每个知识点或章节末尾，提供 2-5 道具有深度的真实面试题。

### 第四步：生成现代化多文件文档网站

摒弃单文件 HTML 的做法，按内容或章节将文档拆分为多个文件，构建一个现代化的静态网站结构。

**技术栈与依赖建议（通过 CDN 引入）：**
- **样式**：Tailwind CSS (`<script src="https://cdn.tailwindcss.com"></script>`)，用于快速构建现代、整洁的 UI。
- **交互**：Alpine.js (`<script defer src="https://cdn.jsdelivr.net/npm/alpinejs@3/dist/cdn.min.js"></script>`)，用于实现侧边栏导航、折叠面板等轻量级交互。
- **图标与高亮**：引入 Lucide Icons 和 Prism.js/Highlight.js 提升阅读体验。

**文件结构要求：**
- `index.html`：网站首页，包含全局目录导航、个人技能概览和学习路线图。
- `chapters/`：按大类或章节拆分的多个 HTML 文件（如 `chapter1_basics.html`, `chapter2_backend.html` 等）。
- 确保各个 HTML 文件之间有清晰的侧边栏导航（Sidebar）进行相互跳转。

## 内容编写规范

### 知识点教学格式

每个知识点（含前置知识）的讲解必须详尽，包含以下结构：

1. **是什么** — 清晰、准确的一句话定义。
2. **为什么重要** — 在实际开发中的作用，解决了什么痛点。
3. **核心原理** — 用通俗易懂但专业的语言深度剖析底层机制和工作原理。
4. **前置知识** — 明确指出理解该知识点需要先掌握什么，并在此处或独立章节展开详细讲解。
5. **实际应用** — 结合简历中对应的项目经验或常见业务场景，说明该技术是如何落地的。

### 面试题格式

每道面试题必须包含：
- **题目**：模拟真实面试官的问法（包含基础题和场景题）。
- **参考答案**：详尽且有条理的解答，突出关键点和底层逻辑。
- **追问方向**：列出面试官可能根据该回答继续深挖的 1-2 个问题。

## 视觉与 UI 设计规范

### 设计目标
打造一个**现代化、高信息密度的工具型 UI**。界面应接近一个会被长期使用的专业技术手册（如官方文档、VitePress 风格），而非花哨的展示型页面。

### 允许与推荐使用
- **轻量级组件**：使用 Tailwind 构建清晰的卡片、警告框（Alerts）、代码块、面包屑导航等。
- **色彩搭配**：纯色或低饱和度的背景色，配合少量的主题强调色突出关键操作和链接。
- **排版与层级**：使用适度的阴影和清晰的边框建立视觉层级关系。充分利用横向空间，保持偏高的信息密度。
- **统一图标**：使用统一风格的 SVG 图标来点缀章节标题和提示信息。

### 禁止事项
- 禁止将所有内容堆砌在一个庞大的单文件 HTML 中。
- 禁止使用蓝紫渐变、玻璃拟态（Glassmorphism）、高透明模糊、光泽效果等过于炫酷的视觉效果。
- 禁止使用 emoji 图标作为主要视觉元素。
- 禁止卡片堆叠式的统计布局。
- 禁止大面积的装饰性图表。
